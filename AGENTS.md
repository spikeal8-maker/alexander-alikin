# AGENTS.md — правила работы coding-агентов

Этот файл обязателен для любого ИИ-агента, который изменяет репозиторий `spikeal8-maker/alexander-alikin`.

## 1. Назначение репозитория

Публичный репозиторий содержит:

- код персональной цифровой платформы Александра Аликина;
- только утверждённый публичный контент;
- публичные метаданные проектов, фактов и материалов;
- документацию архитектуры, редакционного процесса и выпуска.

Приватный архив находится в `spikeal8-maker/alexander-alikin-vault`. Публичный сайт не читает его напрямую.

## 2. Источники истины

Перед работой прочитать:

1. GitHub Issue текущей задачи;
2. `docs/SITE_EXECUTION_MANIFEST.yaml`;
3. `docs/CANONICAL_SITE_STRATEGY.md`;
4. `docs/PAGE_SPECIFICATIONS.md`;
5. `docs/IMPLEMENTATION_ROADMAP.md`;
6. относящиеся к задаче ADR в `docs/DECISIONS/`.

При конфликте приоритет имеют:

1. явно заданный scope текущей Issue;
2. принятые ADR;
3. `CANONICAL_SITE_STRATEGY.md`;
4. `PAGE_SPECIFICATIONS.md`;
5. `IMPLEMENTATION_ROADMAP.md`;
6. остальные документы.

## 3. Модель выполнения

- Одна Issue = один проверяемый пользовательский или технический результат.
- Одна задача = одна ветка = один Draft Pull Request.
- Не изменять `main` напрямую.
- Не начинать следующую задачу до merge и owner review текущей зависимости.
- Не расширять scope ради «заодно улучшить».
- Не выполнять автоматический merge.
- Не закрывать Issue без фактического выполнения Acceptance Criteria.

## 4. Честность результата

Запрещено:

- сообщать PASS без запуска проверки;
- создавать фиктивные скриншоты, URL, метрики или результаты;
- скрывать FAIL, BLOCKED, предупреждения или незапущенные тесты;
- заменять реальную реализацию заглушкой и называть задачу выполненной;
- публиковать неподтверждённые цифры как факты.

В отчёте указывать точные команды, exit code, commit SHA и оставшиеся ограничения.

## 5. Безопасность и публичность

Не коммитить:

- `.env` и локальные конфигурации с секретами;
- токены, пароли, API-ключи, приватные ключи;
- содержимое приватного vault;
- договоры, платёжные данные и чувствительные документы;
- фотографии детей, клиентов и третьих лиц без подтверждённого разрешения;
- исходные ZIP-архивы второго мозга;
- персональные данные посетителей в URL или аналитике.

Секреты хранятся только в предназначенных для этого настройках или вне Git.

## 6. Размер и модульность файлов

Цель — обозримый diff и поддерживаемая архитектура.

- Не коммитить `node_modules`, `dist`, `.astro`, coverage, cache, minified bundles и generated build output.
- Не коммитить base64-архивы и другие непросматриваемые payload после завершения bootstrap.
- Не создавать hand-written source-файлы более 400 строк без обоснования.
- При приближении компонента или страницы к 300–400 строкам разделить её на именованные секции и компоненты.
- Страница собирает секции; контент хранится в collections/data, а не в гигантских массивах внутри компонента.
- Общие цвета, отступы, типографика и состояния задаются токенами, а не копируются по файлам.
- Не дублировать большие тексты между README, Issue и документацией. Ссылаться на канонический документ.
- Нормативный документ может быть длиннее исходного файла, если он остаётся структурированным и не дублирует другой документ.

## 7. Архитектурные ограничения сайта

- Основной контент должен читаться без клиентского JavaScript.
- Интерактивность подключается прогрессивно и только при реальной пользе.
- Статический контент находится в Astro Content Collections или структурированных data-файлах.
- Главная маршрутизирует намерение; бизнес и образование имеют отдельные посадочные страницы.
- Приватный vault не является runtime-зависимостью.
- Obsidian-граф не размещается на первом экране.
- Не использовать сгенерированное лицо, неоновые AI-шаблоны и неподтверждённые счётчики.

## 8. Адаптивность и доступность

Для пользовательских интерфейсов обязательны:

- самостоятельная мобильная компоновка, а не только уменьшенный desktop;
- клавиатурная навигация;
- видимый `focus-visible`;
- корректная семантика заголовков и landmarks;
- alt-тексты для содержательных изображений;
- поддержка `prefers-reduced-motion`;
- отсутствие горизонтальной прокрутки при обычном контенте;
- проверка при 200% zoom;
- текстовые статусы, а не передача смысла только цветом.

## 9. Git и Pull Request

До изменения:

```text
git status --short
git branch --show-current
git log -1 --oneline
```

Если рабочее дерево содержит неизвестные изменения — остановиться и сообщить `BLOCKED`, не стирать их.

Перед push:

```text
git diff --check
git status --short
```

Коммиты должны быть небольшими, содержательными и соответствовать текущей задаче. Не использовать force push без явного разрешения владельца.

## 10. Формат итогового отчёта

```text
TASK:
STATUS: PASS | BLOCKED | FAIL
BRANCH:
HEAD:
VISIBLE_RESULT:
FILES_CHANGED:
TESTS_RUN:
TEST_RESULTS:
SCREENSHOTS:
SECURITY_CHECKS:
KNOWN_LIMITATIONS:
PR:
ISSUE:
NEXT_ALLOWED_TASK:
```

`NEXT_ALLOWED_TASK` не является разрешением начать следующую задачу в той же сессии. После отчёта агент останавливается.
