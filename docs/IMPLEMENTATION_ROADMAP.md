# Канонический план реализации сайта

## Назначение

Этот документ превращает стратегию в последовательную программу разработки. Он определяет порядок, зависимости, видимый результат и критерии приёмки каждого этапа. Этапы обозначают порядок, а не календарные обещания.

## Конечный пользовательский результат версии 1.0

```text
человек находит Александра по имени
→ за 10–20 секунд понимает личность и текущую роль
→ выбирает маршрут: бизнес / образование / знакомство
→ видит релевантные проекты и доказательства
→ читает материал или кейс
→ понимает способ работы и ограничения
→ начинает предметный разговор с сохранённым контекстом
```

## Жёсткие правила программы

1. Одна задача разработки должна давать один проверяемый пользовательский результат.
2. Новый этап не начинается, пока его обязательная зависимость не принята или явно не переведена в допустимый статус.
3. `main` соответствует опубликованной версии; работа ведётся в отдельных ветках и Draft Pull Request.
4. Непроверенный контент не считается временным допустимым наполнением.
5. Макет не принимается без мобильной версии и состояния клавиатурной навигации.
6. Форма не подключается без реального обработчика и политики данных.
7. GitHub Pages не считается запуском, пока не проверены домен, HTTPS, canonical, sitemap и основные маршруты.
8. Интерактивность не блокирует индексируемый статический контент.
9. Метрика успеха — содержательный маршрут и квалифицированное действие, а не количество визуальных эффектов.

- `HOMEPAGE_BLUEPRINT.md`;
- `USER_JOURNEYS_AND_CTA.md`;
- `VISUAL_SYSTEM.md`;

- пользовательские сценарии перечислены;
- документ включён в foundation PR.

### Статус

`in_review` в Draft PR foundation.

## Этап 1. Контентный и доказательный фундамент

### Task ID

`SITE-CONTENT-001`

### Зависимости

- `SITE-STRATEGY-001`;
- приватный vault foundation;
- решения владельца по публичности.

### Видимый результат

Сайт получает реальные, утверждённые материалы вместо общих заглушек.

### Работы

#### Личность

- биография 50 слов;
- биография 150 слов;
- биография 800–1200 слов;
- временная линия;
- текущая профессиональная формула;
- принципы;
- темы выступлений;
- актуальный блок «Сейчас».

#### Медиа

- основной портрет;
- рабочий портрет;
- фото у проекта;
- фото преподавания или выступления;
- архивные фотографии;
- web-копии и social card;
- права, подписи и alt-тексты.

#### Доказательства

- заполнить приоритетные CLAIM/SOURCE/PERMISSION;
- утвердить безопасные публичные формулировки;
- подготовить страницу фактов;
- проверить текущую должность, проекты и контакты.

#### Кейсы

- ИЗО АСА;
- инженерное образование и робототехника;
- ИИ или автоматизация конкретного процесса.

#### Журнал

- стартовые истории;
- мысли;
- системные статьи;
- фактические новости;
- видео с редакционным описанием.

### Приёмка

- нет непроверенных крупных чисел;
- у каждого медиа есть разрешённое использование;
- три кейса имеют контекст, роль, результат, ограничение и источник;
- биографии не противоречат друг другу;
- блок «Сейчас» имеет дату;
- материалы готовы к отдельным content PR.

### Блокеры

- неизвестные права на фотографии;
- неподтверждённая текущая роль;
- отсутствие источников сильных утверждений;
- неразделённые личные и публичные сведения.

# 2. Порядок веток и Pull Request

Рекомендуемая последовательность после foundation:

```text
agent/site-content-baseline
→ agent/site-platform
→ agent/site-design-system
→ agent/site-home
→ agent/site-identity
→ agent/site-business
→ agent/site-education
→ agent/site-projects
→ agent/site-journal
→ agent/site-discovery
→ agent/site-contact
→ agent/site-launch
```

Контентные PR могут идти параллельно техническим только тогда, когда меняют независимые Markdown/медиа-файлы и не создают конфликтов с контентной схемой.

## Правило размера PR

Один PR должен иметь один видимый результат. Не объединять в один PR:

- полный редизайн;
- все тексты;
- форму;
- домен;
- аналитику;
- интерактивный граф.

Foundation PR может быть крупнее, поскольку он фиксирует начальный контракт и bootstrap.

# 3. Definition of Done для пользовательской страницы

- указан Task/Issue;
- реализован утверждённый пользовательский flow;
- контент утверждён;
- ссылки работают;
- CTA ведёт к существующему продолжению;
- desktop/mobile проверены;
- клавиатура и focus проверены;
- alt/labels заполнены;
- SEO metadata заполнены;
- analytics events не содержат PII;
- нет неподтверждённых утверждений;
- build/test PASS;
- screenshots приложены к PR;
- PR остаётся Draft до owner review;
- следующая задача не начинается автоматически.

# 4. Definition of Done для контента

Материал завершён, если:

- имеет аудиторию и цель;
- написан как самостоятельная публичная единица;
- факты имеют источники;
- медиа имеют права;
- указаны дата, автор, статус и review date;
- есть связи;
- CTA соответствует контексту;
- предпросмотр проверен;
- Александр утвердил публикацию.

# 5. Набор обязательных проверок

Минимальные проверки репозитория:

- форматирование;
- lint;
- typecheck;
- content schema;
- internal links;
- external link report;
- build;
- sitemap/RSS;
- draft leakage;
- secret scan;
- accessibility smoke;
- browser navigation smoke;
- mobile screenshot;
- structured data validation;
- noindex/canonical audit.

Нормативная команда должна быть одна, например:

```text
npm run quality
```

Она не должна сообщать PASS, если обязательная проверка фактически не запускалась.

# 6. Решения владельца, необходимые по ходу программы

- канонический домен;
- окончательная формула первого экрана;
- выбор и публикация портрета;
- утверждение биографий;
- разрешение на кейсы и цифры;
- форматы сотрудничества;
- операционный канал входящих;
- аналитическая система;
- правила публикации фотографий детей и клиентов;
- открытость публичного GitHub-репозитория как части образа;
- необходимость комментариев;
- необходимость дополнительных языков.

Отсутствие решения не блокирует всю программу: соответствующий этап остаётся отдельным blocked item, а независимые этапы продолжаются.

# 7. Итоговая последовательность ценности

```text
Стратегия
→ реальные материалы и доказательства
→ технический каркас
→ дизайн-система
→ главная и маршруты
→ личность и репутация
→ бизнес и образование
→ проекты и кейсы
→ журнал
→ поиск и второй мозг
→ контактный процесс
→ SEO, аналитика и запуск
```

Эта последовательность защищает проект от двух типичных ошибок: сначала сделать красивую пустую оболочку либо сначала выгрузить весь архив без структуры и цели.
