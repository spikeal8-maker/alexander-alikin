# SEO, аналитика и эксплуатация сайта

## Назначение

Этот документ определяет, как сайт становится каноническим источником по имени Александра, получает тематическую видимость, измеряет полезные пользовательские маршруты и сохраняет актуальность после запуска.

## Главный принцип

SEO не является набором ключевых слов после разработки. Оно следует из согласованной личности, структуры страниц, доказательств, внутренних связей, технической доступности и регулярного обновления. Аналитика не должна собирать максимум данных; она должна отвечать, понимают ли посетители Александра и переходят ли к содержательному действию.

---

# 1. Поисковые задачи

## Задача A. Брендовая выдача

Сайт должен стать основным ответом по запросам:

- Александр Аликин;
- Александр Сергеевич Аликин;
- Аликин Александр Сергеевич;
- Александр Аликин проекты;
- Александр Аликин предприниматель;
- Александр Аликин педагог;
- Александр Аликин робототехника;
- Александр Аликин искусственный интеллект;
- Александр Аликин ИЗО АСА.

Эти формулировки являются целевой картой страниц, а не требованием многократно повторять имя в каждом абзаце.

## Задача B. Проектная выдача

Каждый значимый проект получает собственный индексируемый URL, который отвечает на:

- что это;
- кто автор или участник;
- какую роль выполнял Александр;
- текущий статус;
- где демонстрация;
- какие материалы связаны.

## Задача C. Тематическая выдача

Журнал и посадочные страницы постепенно формируют видимость по содержательным темам:

- ИИ и автоматизация процессов;
- ИИ-пилот для малого или образовательного бизнеса;
- база знаний и ИИ-ассистент;
- проектное инженерное обучение;
- робототехника и исследовательские проекты;
- ИИ для педагога;
- разработка образовательной программы;
- обучение через проект и ошибку.

Тематические страницы создаются только при наличии самостоятельного полезного материала и релевантной практики.

---

# 2. Каноническая сущность человека

## Единые данные

На всех управляемых площадках согласуются:

- `Александр Сергеевич Аликин`;
- короткое описание;
- портрет;
- основной домен;
- текущая профессиональная формула;
- официальные ссылки;
- место текущей деятельности только при подтверждённой актуальности.

## Варианты имени

В `<title>`, H1, биографии и структурированных данных используется полное имя. В естественном тексте допустимо `Александр Аликин`. Варианты написания и транслитерации фиксируются в метаданных и связанных профилях, но не создают искусственные повторения.

## Страница-источник

`/about/` является канонической биографией. Главная — каноническим входом. `/facts/` — канонической страницей публичных подтверждений. Социальные профили ссылаются на основной домен.

# 3. URL-структура

Рекомендуемая структура:

```text
/
/about/
/business/
/education/
/projects/
/projects/<slug>/
/journal/
/journal/stories/<slug>/
/journal/thoughts/<slug>/
/journal/articles/<slug>/
/journal/news/<slug>/
/now/
/second-brain/
/facts/
/press/
/collaboration/
/contacts/
/search/
/privacy/
```

## Правила slug

- латиница;
- короткое понятное название;
- дефисы;
- без даты в URL, если дата не является смысловой частью;
- URL не меняется из-за редактирования заголовка;
- старый URL получает постоянный redirect;
- один материал имеет один индексируемый canonical URL.

## Фильтры

Фильтры проектов и журнала должны сохранять состояние для пользователя, но не создавать бесконечное количество индексируемых комбинаций. Каноническими являются основные рубрики и материалы; технические комбинации фильтров получают canonical на базовую страницу либо `noindex` по принятой реализации.

---

# 4. Метаданные страницы

Каждая индексируемая страница содержит:

- уникальный `title`;
- уникальный `description`;
- один H1;
- canonical URL;
- Open Graph title/description/image;
- тип Open Graph;
- дату публикации и обновления при наличии;
- автора;
- язык;
- robots directive;
- структурированные данные по фактическому типу.

## Пример title

```text
Александр Аликин — инженер, педагог и предприниматель
```

## Пример description

```text
Официальный сайт Александра Аликина: технологические и образовательные проекты, ИИ и автоматизация, биография, статьи, истории и форматы сотрудничества.
```

Фактический текст утверждается отдельно и не должен превышать смысловую ёмкость ради формального лимита.

# 5. Структурированные данные

## Главная и «Обо мне»

- `Person`;
- `ProfilePage` на странице биографии;
- `sameAs` только для официальных проверенных профилей;
- `knowsAbout` используется умеренно и отражает реальные области;
- `worksFor` только при актуальном и разрешённом подтверждении;
- `alumniOf`, награды и должности не добавляются без источников.

## Материалы

- `Article` или более точный подтип при соответствии;
- автор;
- дата публикации;
- дата изменения;
- изображение;
- headline;
- canonical URL.

## Видео

- `VideoObject` при наличии фактических данных: название, описание, thumbnail, uploadDate, embed/content URL.

## Навигация

- `BreadcrumbList` на вложенных страницах.

## Проекты

Выбирается фактически корректный тип: `CreativeWork`, `SoftwareApplication`, `Course` или общий `WebPage`. Нельзя назначать тип только ради расширенного результата.

## Правило

Структурированные данные не содержат фактов, которых нет в видимом содержимом страницы.

# 6. Внутренние связи

## Главная модель

```text
Личность
↔ проекты
↔ материалы
↔ факты
↔ сотрудничество
```

## Проект связывается с

- биографическим этапом;
- кейсом;
- статьями;
- историями;
- видео;
- фактами;
- контекстным CTA.

## Материал связывается с

- проектом-источником;
- серией;
- предыдущим и следующим смысловым материалом;
- релевантной посадочной страницей.

## Запрет

Не создавать блок «Вам также может понравиться» только на основе случайных общих тегов. Ручная редакционная связь имеет приоритет.

# 7. Изображения и медиа

## Файлы

- семантические имена;
- WebP/AVIF с безопасным fallback по выбранной реализации;
- width/height для предотвращения скачков layout;
- responsive `srcset`;
- отдельный кадр social card;
- alt-текст;
- удалённые лишние метаданные;
- оригиналы не хранятся в публичной web-папке без необходимости.

## Видео

Тяжёлый внешний embed не загружается до взаимодействия либо используется лёгкий preview. Первый экран не зависит от внешнего видеоплеера.

# 8. Sitemap, robots и RSS

## Sitemap

Включает:

- главные страницы;
- опубликованные проекты;
- опубликованные материалы;
- страницы фактов и прессы при готовности.

Не включает:

- draft;
- preview;
- технические filter URLs;
- внутренний поиск;
- страницы с закрытым статусом.

## Robots

`robots.txt` указывает основной sitemap и не используется как механизм защиты приватных данных. Приватный материал вообще не попадает в публичную сборку.

## RSS

RSS содержит журнал или отдельные потоки по принятым рубрикам. Каждая запись ведёт на каноническую страницу сайта и имеет корректную дату.

---

# 9. Домены и редиректы

## Канонический домен

Выбирается один персональный домен. Проектные домены не становятся основным адресом личности без отдельного решения.

## GitHub Pages URL

Технический адрес остаётся способом развёртывания, но не должен создавать индексируемый дубликат канонического сайта. Конфигурация `site`, canonical и домена должна быть единой.

## Существующие домены

Перед переносом проводится инвентарь:

- владелец;
- DNS;
- текущие страницы;
- внешние ссылки;
- поисковый трафик;
- необходимость redirect;
- срок сохранения.

## Redirect map

В репозитории хранится таблица:

```text
old_url → new_url → reason → verified_at
```

URL не удаляется без решения о redirect или статусе 410 для действительно удалённого контента.

# 10. Поисковая базовая линия

До запуска фиксируются:

- результаты первой страницы по полному имени;
- какие результаты управляются Александром;
- какие старые профили доминируют;
- индексируемые проектные домены;
- ошибки написания имени;
- текущие title/description внешних профилей;
- доступность официальных страниц.

После запуска изменения сравниваются с базовой линией, а не оцениваются по впечатлению.

# 11. Аналитическая модель

## Главные вопросы

1. Понимают ли посетители, кто такой Александр?
2. Какой маршрут выбирают?
3. Видят ли релевантное доказательство до контакта?
4. Какие материалы создают второй просмотр или возврат?
5. Какие страницы приводят к содержательным обращениям?
6. Где человек останавливается или возвращается назад?

## Основные KPI

### Доверие и идентичность

- доля переходов с главной к `about`, `projects` или `facts`;
- открытия внешних источников;
- результаты сценарных интервью;
- доля управляемых результатов брендовой выдачи.

### Маршрутизация

- выбор business/education/about;
- переход к релевантному кейсу;
- возврат с посадочной страницы на главную как возможный сигнал тупика;
- успешное достижение CTA.

### Контент

- второй просмотр материала;
- переход по смысловой связи;
- открытие проекта из статьи;
- RSS/Telegram follow click;
- возвращающиеся посетители только при допустимой privacy-модели.

### Сотрудничество

- начало брифа;
- завершение брифа;
- тип обращения;
- источник маршрута;
- наличие конкретного процесса или программы;
- квалификация обращения вручную.

### Техническое качество

- ошибки 404;
- битые внешние ссылки;
- Core Web/field metrics при наличии данных;
- ошибки формы;
- неуспешные deploy;
- просроченные страницы.

## Vanity metrics

Не используются как главный критерий:

- общее число просмотров;
- количество опубликованных постов;
- число соцсетей;
- глубина страницы без понимания намерения;
- время на странице без контекста.

# 12. События аналитики

## Без PII

События не содержат:

- имя посетителя;
- email;
- телефон;
- полный текст обращения;
- URL с персональными параметрами;
- содержание закрытого проекта.

## Набор событий

### Навигация

```text
nav_click
home_route_select
search_open
search_result_open
```

### Доказательства

```text
project_open
case_open
evidence_open
external_source_open
demo_open
repository_open
```

### Контент

```text
content_open
content_depth_50
content_depth_90
related_content_click
second_content_view
rss_click
channel_click
```

### Сотрудничество

```text
collaboration_type_select
brief_start
brief_submit
brief_error
contact_channel_click
press_request_start
press_request_submit
```

## Свойства

Допустимы:

- page_type;
- route_type;
- content_type;
- project_slug;
- topic;
- CTA ID;
- source_page;
- device class;
- build version.

Не передаётся пользовательский текст.

# 13. Выбор аналитической системы

Решение принимается до подключения скрипта. Критерии:

- соответствие юрисдикции и политике данных;
- минимизация cookies;
- возможность работать без рекламного профилирования;
- поддержка кастомных событий;
- экспорт данных;
- стоимость владения;
- контроль над сроком хранения;
- отсутствие блокировки основной страницы при сбое.

До выбора система проектируется через единый внутренний analytics adapter, чтобы страницы не зависели от конкретного поставщика.

# 14. Качественные исследования

Аналитика не заменяет наблюдение за людьми.

## Перед запуском

Минимум пять сценарных тестов:

- поиск по имени;
- бизнес-задача;
- образовательная задача;
- партнёрский проект;
- СМИ;
- чтение связанного материала.

## Вопросы после теста

- кто такой Александр;
- чем он отличается;
- где находится нужная страница;
- чему доверяет пользователь;
- какое действие ожидает после кнопки;
- какие слова непонятны;
- что выглядит неподтверждённым;
- что отсутствует для решения.

Результаты превращаются в Issues, а не остаются устными замечаниями.

# 15. Эксплуатационный цикл

## Еженедельно

- обработать входящие Issues;
- проверить фактические новости;
- обновить «Сейчас» при изменении;
- проверить failed deploy и ошибки формы;
- просмотреть новые 404.

## Ежемесячно

- опубликовать утверждённый крупный материал при наличии;
- обновить проект;
- проверить внешние ссылки;
- проверить истекающие факты;
- посмотреть путь от входа к CTA;
- сформировать короткий операционный отчёт.

## Ежеквартально

- брендовая поисковая выдача;
- актуальность биографии и press-пакета;
- пересмотр аудиторий и реальных обращений;
- аудит безопасности и зависимостей;
- проверка резервного восстановления;
- обновление roadmap;
- закрытие неработающих контентных серий.

## После значимого события

- создать news item;
- обновить проект/факт;
- обновить `now`;
- проверить ссылки внешних профилей;
- при необходимости обновить press-пакет.

# 16. Владение актуальностью

Каждая ключевая страница содержит в метаданных:

```yaml
owner: "Alexander Alikin"
lastVerifiedAt: "2026-07-19"
reviewAt: "2026-10-19"
status: published
```

Дата `reviewAt` зависит от типа:

- контакты и текущая роль — короткий цикл;
- `now` — очень короткий цикл;
- проект active — регулярный цикл;
- архивная история — длинный цикл;
- факт — по характеру утверждения;
- правовая страница — при изменении сервисов или процесса.

Просроченная страница создаёт Issue или отчёт проверки. Она не удаляется автоматически, но теряет метку актуальности до пересмотра.

# 17. Release и rollback

## Выпуск

```text
feature/content branch
→ Draft PR
→ quality checks
→ owner review
→ ready for review
→ merge to main
→ GitHub Actions deploy
→ production smoke
→ release note
```

## Smoke после deploy

- главная;
- about;
- business;
- education;
- project;
- article;
- collaboration;
- search;
- sitemap;
- RSS;
- 404;
- canonical domain;
- mobile navigation.

## Rollback

- определить последний исправный commit;
- revert проблемного PR либо повторно развернуть исправный commit по принятой процедуре;
- не изменять вручную production-файлы вне Git;
- зафиксировать причину и проверку в Issue;
- обновить контент, если ошибка связана с фактом или правами.
